# Story 1.2 — Migrar Backend para Azure Web App

> **Epic:** [EPIC-001 — VM-to-WebApp Modernization](../epics/EPIC-001-vm-to-webapp-modernization.md)
> **Status:** Draft
> **Estimated:** 30 min
> **Pré-requisito:** [Story 1.1](1.1.story.md) concluída (3 VMs rodando)

---

## Story

**Como** aluno do evento "Copa do Mundo Azure",
**eu quero** substituir a `vm-back` por um **Azure Web App for Windows** (`fifa2026-back`),
**para que** eu deixe de gerenciar uma VM de backend (sem mais patches de OS, IIS instalado pra mim, deploy via push) e veja na prática o ganho operacional do PaaS — mantendo o frontend e o banco em VMs por enquanto, para isolar a mudança.

---

## Contexto

Estado **antes** desta story:
```
Internet ──▶ vm-front (IIS) ──▶ vm-back (iisnode :3001) ──▶ vm-db (SQL Server)
```

Estado **depois** desta story:
```
Internet ──▶ vm-front (IIS) ──▶ fifa2026-back (Azure Web App) ──▶ vm-db (SQL Server)
                                       │
                                       └─ outbound via VNet integration ──▶ vm-db
```

A `vm-back` pode ser **desligada** ao final desta story.

---

## Acceptance Criteria

1. **AC-1** — Azure Web App `fifa2026-back` provisionada (Windows, Node 18, plano B1).
2. **AC-2** — App Settings configuradas: `DB_SERVER` apontando para o **IP privado** ou hostname de `vm-db`, mais `DB_USER`, `DB_PASSWORD`, `DB_NAME`, `JWT_SECRET`, `FRONTEND_URL`.
3. **AC-3** — Web App tem **VNet Integration** habilitada na VNet da Story 1.1, permitindo o backend alcançar a `vm-db` no IP privado.
4. **AC-4** — Backend deployado no Web App via GitHub Actions (`deploy-backend.yml`) ou manualmente via `az webapp deploy`.
5. **AC-5** — `web.config` da `vm-front` editado: `API Proxy` agora aponta para `https://fifa2026-back.azurewebsites.net` em vez de `http://<IP_BACK>:3001`.
6. **AC-6** — **Smoke test passa:** acesso à app via IP público da `vm-front` continua funcionando (login + listar jogos).
7. **AC-7** — `vm-back` pode ser desligada (`az vm deallocate`) sem afetar a app.

## Tasks / Subtasks

### Provisionar `fifa2026-back` (10 min)

- [ ] No mesmo grupo de recursos da story anterior (ou em um novo `fifa2026-paas-rg`):
  ```bash
  az appservice plan create -g fifa2026-paas-rg -n fifa2026-plan --sku B1 --is-linux false
  az webapp create -g fifa2026-paas-rg -p fifa2026-plan -n fifa2026-back --runtime "node:18LTS"
  ```
- [ ] HTTPS Only + TLS 1.2 + FTPS Disabled:
  ```bash
  az webapp update -g fifa2026-paas-rg -n fifa2026-back --https-only true
  az webapp config set -g fifa2026-paas-rg -n fifa2026-back --ftps-state Disabled --min-tls-version 1.2
  ```

> **Atalho:** ou rode o módulo `infra/modules/web-app-backend.bicep` (ou `provision.sh` passos 2 e 4).

### Configurar VNet Integration (5 min)

- [ ] Habilitar VNet Integration apontando para a `subnet-app` da Story 1.1:
  ```bash
  az webapp vnet-integration add \
    -g fifa2026-paas-rg -n fifa2026-back \
    --vnet vnet-fifa2026 --subnet subnet-app
  ```

> Sem isso, o Web App não consegue alcançar `vm-db` no IP privado.

### App Settings com credenciais do banco (5 min)

- [ ] Pegar IP privado da `vm-db` da Story 1.1 (`IP_DB`)
- [ ] Configurar:
  ```bash
  az webapp config appsettings set -g fifa2026-paas-rg -n fifa2026-back --settings \
    DB_SERVER="<IP_DB>" \
    DB_PORT=1433 \
    DB_USER=fifa2026_db \
    DB_PASSWORD='<senha>' \
    DB_NAME=FIFA2026Tickets \
    JWT_SECRET='<string aleatória>' \
    JWT_EXPIRES_IN=7d \
    FRONTEND_URL="http://<IP_PUBLICO_FRONT>" \
    WEBSITE_NODE_DEFAULT_VERSION="~18"
  ```

### Deploy do código backend (5 min)

**Opção A — Manual (zip deploy):**
- [ ] Local: `cd fifa2026-api && zip -r ../api.zip . -x "logs/*" "node_modules/*" ".env"`
- [ ] `cd FIFA2026-APP && az webapp deploy -g fifa2026-paas-rg -n fifa2026-back --src-path api.zip --type zip`

**Opção B — GitHub Actions:**
- [ ] Setup secret `AZURE_BACKEND_PUBLISH_PROFILE` (ver `.github/workflows/README.md`)
- [ ] Trigger workflow `Deploy Backend (FIFA 2026 API)` manualmente

### Reapontar frontend (3 min)

- [ ] RDP na `vm-front`
- [ ] Editar `C:\inetpub\wwwroot\fifa2026-web\web.config`:
  - **Antes:** `<action type="Rewrite" url="http://<IP_BACK>:3001/api/{R:1}" />`
  - **Depois:** `<action type="Rewrite" url="https://fifa2026-back.azurewebsites.net/api/{R:1}" />`
- [ ] Reiniciar IIS: `iisreset` no PowerShell admin

### Smoke test (2 min)

- [ ] Browser → `http://<IP_PUBLICO_FRONT>` → login + listar jogos
- [ ] Health do backend direto: `curl https://fifa2026-back.azurewebsites.net/api/health/db` deve retornar `{ status: "ok", database: "connected" }`
- [ ] (Opcional) Desligar `vm-back`: `az vm deallocate -g fifa2026-vm-rg -n vm-back`. App continua funcionando.

## Dev Notes

**Por que VNet Integration e não public IP no SQL Server?**
- VNet Integration mantém o tráfego **dentro do VNet** (não vai para a Internet)
- Se você expusesse `vm-db` com IP público, qualquer um do mundo bateria na porta 1433
- VNet Integration é a forma "barata" de conectar Web App a recursos privados sem provisionar Application Gateway ou Front Door

**B1 + VNet Integration:** SKU B1 do App Service Plan **suporta** VNet Integration regional. Se você estivesse em F1 (Free), não suportaria.

**Por que ainda mantém o frontend e o DB em VM?**
- **Princípio**: cada story muda **uma** coisa. Migrar 3 camadas de uma vez torna debug impossível.
- Você verá o Web App rodando em produção real **com** uma VM de DB ainda ativa — isso é um padrão híbrido comum em migrações reais.

**Custo desta story:**
- App Service Plan B1: ~$13/mês (rodando 24h)
- Web App em si: $0 (incluso no plan)
- VNet Integration: $0
- Total marginal: $13/mês. Apague o RG ao final do evento.

## Rollback

Se algo der errado:

1. **Reverter o `web.config` da vm-front** para o IP privado do backend antigo:
   ```xml
   <action type="Rewrite" url="http://<IP_BACK>:3001/api/{R:1}" />
   ```
2. Religar `vm-back` se ela foi desligada: `az vm start -g fifa2026-vm-rg -n vm-back`
3. (Opcional) Apagar `fifa2026-back`: `az webapp delete -g fifa2026-paas-rg -n fifa2026-back`

App volta ao estado da Story 1.1.
