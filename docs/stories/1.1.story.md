# Story 1.1 — Deploy inicial em 3 VMs

> **Epic:** [EPIC-001 — VM-to-WebApp Modernization](../epics/EPIC-001-vm-to-webapp-modernization.md)
> **Status:** Draft
> **Estimated:** 45 min
> **Pré-requisito:** Subscription Azure ativa, az CLI instalado, conta TFTEC

---

## Story

**Como** aluno do evento "Copa do Mundo Azure" da TFTEC,
**eu quero** provisionar e deployar a aplicação FIFA 2026 em **3 Máquinas Virtuais separadas** (frontend público, backend privado, banco de dados privado),
**para que** eu entenda o estado inicial "tradicional" antes de começar a modernização para PaaS, e veja na prática por que separar camadas em VMs traz isolamento mas também complexidade operacional.

---

## Contexto

Este é o **estado de partida** da nossa jornada. No final desta story você terá:

```
Internet ──▶  VM-Front (pública, IIS) ──▶ VM-Back (privada, iisnode :3001) ──▶ VM-DB (privada, SQL Server :1433)
```

Os próximos 3 stories vão **substituir uma VM de cada vez** por um serviço PaaS, sempre mantendo o app funcional. Hoje você vê todas as 3 VMs.

---

## Acceptance Criteria

1. **AC-1** — 3 VMs provisionadas no Azure: `vm-front` (Windows Server, IP público), `vm-back` (Windows Server, sem IP público), `vm-db` (Windows Server, sem IP público), todas no mesmo VNet.
2. **AC-2** — NSGs configuradas: `vm-front` aceita 80/443 da Internet; `vm-back` aceita 3001 só de `vm-front`; `vm-db` aceita 1433 só de `vm-back`.
3. **AC-3** — Em `vm-db`: SQL Server instalado, banco `FIFA2026Tickets` restaurado a partir do `FIFA2026Tickets.bacpac`.
4. **AC-4** — Em `vm-back`: backend Node.js publicado em IIS via iisnode na porta 3001, com `.env` apontando para o IP privado da `vm-db`.
5. **AC-5** — Em `vm-front`: frontend (build do Lovable) publicado em IIS na porta 80, com `web.config` fazendo reverse proxy `/api/*` para o IP privado de `vm-back:3001`.
6. **AC-6** — **Smoke test passa:** acessar o IP público de `vm-front` no browser → carrega home → login com `admin@fifa2026.com / admin123` → listar jogos retorna dados.
7. **AC-7** — `vm-back` e `vm-db` **não** respondem a chamadas vindas da Internet (testar via `curl` de fora da VNet retorna timeout).

## Tasks / Subtasks

### Pré-flight (5 min)

- [ ] Confirmar `az login` ativo: `az account show`
- [ ] Confirmar quota Azure suficiente para 3 VMs Standard_B2s na sua subscription
- [ ] Definir região: `export LOCATION=eastus`
- [ ] Definir grupo de recursos: `export RG=fifa2026-vm-rg && az group create -n $RG -l $LOCATION`

### Provisionar VNet e VMs (15 min)

- [ ] Criar VNet `vnet-fifa2026` com 1 subnet (`subnet-app`)
- [ ] Criar `vm-front` (Windows Server 2022, Standard_B2s, **com** IP público, NSG: 80/443/3389)
- [ ] Criar `vm-back` (Windows Server 2022, Standard_B2s, **sem** IP público, NSG: 3001 da subnet, 3389 do seu IP)
- [ ] Criar `vm-db` (Windows Server 2022, Standard_B2s, **sem** IP público, NSG: 1433 da subnet, 3389 do seu IP)
- [ ] Anotar IPs privados: `IP_FRONT`, `IP_BACK`, `IP_DB`

### Configurar `vm-db` — SQL Server (10 min)

- [ ] RDP na `vm-db` (via Bastion ou jump host)
- [ ] Instalar SQL Server 2022 Developer Edition (gratuito) com Mixed Authentication
- [ ] Habilitar TCP/IP no SQL Server Configuration Manager → reiniciar serviço
- [ ] Criar regra de firewall do Windows: porta 1433 inbound
- [ ] Restaurar `FIFA2026Tickets.bacpac` via SSMS (ver Dev Notes)
- [ ] Criar login `fifa2026_db` com senha forte → mapear para `FIFA2026Tickets` com `db_datareader` + `db_datawriter`

### Configurar `vm-back` — Backend Node.js (10 min)

- [ ] RDP na `vm-back`
- [ ] Instalar IIS + URL Rewrite + iisnode (ver Dev Notes — link para guia detalhado)
- [ ] Baixar a aplicação para `C:\inetpub\wwwroot\fifa2026-api`
- [ ] Criar `.env` em `fifa2026-api/` com base em `.env.example`, preenchendo:
  - `DB_SERVER=<IP_DB>`
  - `DB_USER=fifa2026_db` / `DB_PASSWORD=<senha que você criou>`
  - `JWT_SECRET=` (gerar string aleatória)
  - `FRONTEND_URL=http://<IP_PUBLICO_FRONT>`
- [ ] Rodar `npm install --omit=dev` em `fifa2026-api/`
- [ ] Criar site no IIS apontando para `fifa2026-api/`, port 3001
- [ ] Testar local na VM: `Invoke-RestMethod http://localhost:3001/api/health`

### Configurar `vm-front` — Frontend (5 min)

- [ ] RDP na `vm-front`
- [ ] Instalar IIS + URL Rewrite + ARR (Application Request Routing)
- [ ] Habilitar proxy no ARR (Server Proxy Settings → Enable)
- [ ] Baixar build do frontend para `C:\inetpub\wwwroot\fifa2026-web`
- [ ] Editar `fifa2026-web/web.config`: na regra `API Proxy`, trocar `http://localhost:3001` por `http://<IP_BACK>:3001`
- [ ] Criar site no IIS apontando para `fifa2026-web/`, port 80

### Smoke test (5 min)

- [ ] No seu browser, acessar `http://<IP_PUBLICO_FRONT>` — home deve carregar
- [ ] Clicar em "Entrar" → login com `admin@fifa2026.com / admin123`
- [ ] Listar jogos → deve mostrar 104 jogos (FIFA 2026 completo — do bacpac regenerado em 2026-05-15)
- [ ] (Opcional) Tentar acessar `http://<IP_BACK>:3001/api/health` direto da Internet → **deve dar timeout** (privado)

> ⚠️ **@po freshness audit 2026-05-15** ([docs/qa/2026-05-15-po-validation-EPIC-001.md](../qa/2026-05-15-po-validation-EPIC-001.md)): contagens atualizadas p/ o bacpac regenerado (104 jogos / 17 estádios / 49 seleções). **Gate D2:** confirmar que `admin@fifa2026.com / admin123` ainda existe no seed novo antes do dry-run.

## Dev Notes

**Roteiro detalhado de IIS+iisnode:** já existe em `Lovable/World Cup Tickets Hub/DEPLOY_IIS_SIMPLIFICADO.md`. **Importante:** esse roteiro original assume front+back na MESMA VM. Nesta story, você aplica os passos do roteiro **separadamente** em `vm-back` (backend) e `vm-front` (frontend).

**Restaurar bacpac via SSMS na VM-DB:**
1. SSMS → Connect → `localhost`
2. Right-click "Databases" → "Import Data-Tier Application"
3. Importar `FIFA2026Tickets.bacpac` → `Local disk`
4. Database name: `FIFA2026Tickets` → OK
5. Aguardar import (~1 min)

**Por que iisnode e não pm2 ou serviço Windows?**
- iisnode integra Node ao IIS como um handler — você ganha logs, restart automático, gestão de pool de aplicação tudo dentro do IIS
- pm2 funcionaria mas adiciona uma segunda ferramenta para o aluno aprender
- iisnode é o padrão da indústria para Node em Windows Server

**Observação sobre custo:** 3 VMs B2s rodando 24h ≈ $90/mês. Para o evento, **desligue as VMs após cada story** (`az vm deallocate`) para reduzir custo proibitivo.

## Validation / Smoke Test

```powershell
# Do seu computador (fora da VNet):
$FRONT_IP = "<IP_PUBLICO_FRONT>"

# 1. Frontend deve responder
Invoke-WebRequest "http://$FRONT_IP" -UseBasicParsing | Select-Object StatusCode

# 2. API via proxy do IIS deve funcionar
Invoke-RestMethod "http://$FRONT_IP/api/health"

# 3. Login
$body = @{ email='admin@fifa2026.com'; password='admin123' } | ConvertTo-Json
$result = Invoke-RestMethod "http://$FRONT_IP/api/auth/login" -Method POST -ContentType 'application/json' -Body $body
$result.user.email   # admin@fifa2026.com

# 4. Listar jogos com token
$headers = @{ Authorization = "Bearer $($result.token)" }
(Invoke-RestMethod "http://$FRONT_IP/api/matches" -Headers $headers).matches.Count   # 104

# 5. Backend NÃO deve responder direto da Internet
$BACK_IP_PRIVADO = "<IP_BACK>"
try {
  Invoke-WebRequest "http://${BACK_IP_PRIVADO}:3001/api/health" -TimeoutSec 5 -UseBasicParsing
  Write-Error "FALHOU: backend está acessível pela Internet!"
} catch {
  Write-Host "OK: backend é privado (timeout esperado)" -ForegroundColor Green
}
```

## Rollback

Se algo der errado e você quiser começar do zero:

```bash
az group delete --name fifa2026-vm-rg --yes --no-wait
```

Isso apaga **tudo** (3 VMs, VNet, NICs, NSGs, discos). Custo zero a partir desse momento.
