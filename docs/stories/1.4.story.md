## Story

**Como** aluno do evento "Copa do Mundo Azure",
**eu quero** substituir a `vm-db` por um **Azure SQL Database** (`fifa2026-sql/FIFA2026Tickets`),
**para que** eu encerre a jornada de modernização com **100% PaaS** — eliminando a última VM, ganhando backups automáticos, alta disponibilidade nativa e patching gerenciado, e aprendendo na prática como **migrar dados** (não só código) entre serviços usando o `.bacpac`.

---

## Contexto

Estado **antes**:
```
Internet ──▶ fifa2026-web (Web App) ──▶ fifa2026-back (Web App) ──▶ vm-db (SQL Server)
```

Estado **depois** (estado final do epic):
```
Internet ──▶ fifa2026-web (Web App) ──▶ fifa2026-back (Web App) ──▶ Azure SQL Database
```

**Toda a infra agora é PaaS.** As 3 VMs originais podem ser apagadas.

**Insight chave:** migração de **compute** (Stories 1.2, 1.3) é tipicamente "redeploy do mesmo código em um novo runtime". Migração de **dados** (esta story) tem etapas específicas: export, transferir, import, reapontar conexão, validar integridade.

## Tasks / Subtasks

### Provisionar Azure SQL (5 min)

- [ ] Criar SQL Server lógico:
  ```bash
  read -s -p "SQL admin password: " SQL_PWD; echo
  az sql server create \
    -g fifa2026-paas-rg -n fifa2026-sql -l eastus \
    --admin-user fifa2026admin \
    --admin-password "$SQL_PWD" \
    --minimal-tls-version 1.2
  ```
- [ ] Permitir Azure services (Web App acessa via outbound público):
  ```bash
  az sql server firewall-rule create \
    -g fifa2026-paas-rg -s fifa2026-sql -n AllowAllAzureServices \
    --start-ip-address 0.0.0.0 --end-ip-address 0.0.0.0
  ```
- [ ] (Para o import via SSMS / sqlpackage do seu IP, adicione o seu IP):
  ```bash
  MY_IP=$(curl -s https://api.ipify.org)
  az sql server firewall-rule create \
    -g fifa2026-paas-rg -s fifa2026-sql -n MyDevIP \
    --start-ip-address $MY_IP --end-ip-address $MY_IP
  ```

> Atalho: módulo `infra/modules/sql-database.bicep` ou `provision.sh` passo 3.

### Importar `.bacpac` (15 min — depende do tamanho)

**Opção A — via Azure CLI (recomendada):**
```bash
# 1. Subir o bacpac para um storage temporário
STORAGE=fifa2026sa$RANDOM
az storage account create -g fifa2026-paas-rg -n $STORAGE -l eastus --sku Standard_LRS
KEY=$(az storage account keys list -g fifa2026-paas-rg -n $STORAGE --query [0].value -o tsv)
az storage container create --name bacpac --account-name $STORAGE --account-key $KEY
az storage blob upload \
  --account-name $STORAGE --account-key $KEY \
  --container-name bacpac \
  --file FIFA2026Tickets.bacpac \
  --name FIFA2026Tickets.bacpac

# 2. Importar
az sql db import \
  -g fifa2026-paas-rg -s fifa2026-sql -n FIFA2026Tickets \
  --storage-key-type StorageAccessKey --storage-key $KEY \
  --storage-uri "https://$STORAGE.blob.core.windows.net/bacpac/FIFA2026Tickets.bacpac" \
  --admin-user fifa2026admin \
  --admin-password "$SQL_PWD"

# 3. Limpar storage
az storage account delete -g fifa2026-paas-rg -n $STORAGE --yes
```

**Opção B — via portal:** SQL Server → "Import database" → Storage → upload do bacpac → import.

**Opção C — via SSMS (Windows):** Connect ao `fifa2026-sql.database.windows.net` → Right-click "Databases" → "Import Data-Tier Application" → arquivo local.

## Dev Notes

**Por que o bacpac e não SQL replication ou DMS?**
- Bacpac é o formato **portátil** do SQL Server (export schema + dados)
- Funciona em qualquer SQL Server moderno (on-prem, VM, Azure SQL DB, Azure SQL MI)
- Para datasets pequenos (< 200GB), é a forma mais simples
- Para datasets grandes ou migração com **zero downtime**, você usaria Database Migration Service (DMS) — fora do escopo deste evento

**Por que firewall AllowAllAzureServices é "ok" para evento mas não produção?**
- A regra abre o SQL para **qualquer** outbound IP do Azure (qualquer cliente Azure pode tentar conectar)
- A autenticação ainda é exigida — sem `fifa2026admin` + senha, não entra
- **Em produção real**: trocar por **Private Endpoint** + VNet integration → tráfego privado, sem exposição pública. Isto é uma story de "modernização avançada" fora deste epic.

**Por que removemos VNet Integration nesta story?**
- VNet Integration era necessária na 1.2 porque o DB estava em **IP privado** dentro da VNet
- Agora o DB é PaaS, alcançado via outbound público (com firewall)
- Manter VNet Integration sem motivo é overhead de configuração

**Custo final acumulado (após esta story):**
- App Service Plan B1: ~$13/mês
- Azure SQL Database Basic: ~$5/mês
- Total: **~$18/mês** (vs ~$90/mês da Story 1.1 com 3 VMs)

## Rollback

Se a migração falhou ou os dados estão corrompidos:

1. **Reapontar backend para a vm-db de novo:**
   ```bash
   az webapp config appsettings set -g fifa2026-paas-rg -n fifa2026-back --settings \
     DB_SERVER="<IP_DB_VM>" \
     DB_USER="fifa2026_db" \
     DB_PASSWORD="<senha-vm-db>"
   ```
2. **Religar VNet Integration** (necessária para o IP privado):
   ```bash
   az webapp vnet-integration add -g fifa2026-paas-rg -n fifa2026-back \
     --vnet vnet-fifa2026 --subnet subnet-app
   ```
3. **Não apagar `vm-db`** até validar que `fifa2026-sql` está saudável por algumas horas

App volta ao estado da Story 1.3.
